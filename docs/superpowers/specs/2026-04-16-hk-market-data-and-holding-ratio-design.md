# 港股数据源与持仓占比设计

## 背景

当前项目的股票行情统一通过 Finnhub 获取。现状已经确认：

- 当前 Finnhub Key 可以正常获取美股报价
- 当前 Finnhub Key 对港股报价返回 `You don't have access to this resource.`
- 因此港股无法通过现有行情链路获取价格

此外，股票持仓目前没有“市场”字段，系统无法区分一只股票是美股还是港股，也无法根据市场路由到不同的数据源。

用户新增的展示需求是：

- 每个账户中的每个持仓，需要展示“该持仓占当前账户总体的比例”
- 这个“账户总体”需要把流动现金也算进去
- 如果某账户同时包含港股和美股，计算占比时统一换算成美元后再计算

## 目标

- 股票持仓支持区分 `US` / `HK` 市场
- 美股继续使用 Finnhub 报价
- 港股改用 iTick 免费 REST API 报价
- 港股代码支持多种输入形式，并统一规范化存储
- 在账户详情中为每条持仓展示其占当前账户总体的比例
- 占比计算时把流动现金计入账户总体
- 对港美股混合账户，统一换算成美元后再计算比例
- 保持旧数据自动迁移，不要求用户手动重录

## 非目标

- 不接入 A 股或其他市场
- 不实现手动覆盖股票价格
- 不更改基金账户的手动市值录入方式
- 不新增更复杂的收益率、成本价或盈亏功能

## 方案对比

### 方案 A：市场挂在账户上

优点：
- 表单简单
- 报价路由更直观

缺点：
- 一个账户只能是美股或港股
- 无法支持同账户混合持有港美股
- 不满足“港美股混合账户统一换算成美元后计算占比”的需求

### 方案 B：市场挂在每条持仓上（推荐）

示例：

```json
{ "code": "AAPL", "market": "US", "shares": 1 }
{ "code": "2714", "market": "HK", "shares": 100 }
```

优点：
- 同一账户可以混合持有美股和港股
- 可以按持仓市场分流请求不同数据源
- 最符合新增的占比计算需求

缺点：
- 股票录入时多一个市场选择

### 方案 C：完全根据代码自动猜市场

优点：
- 录入最省事

缺点：
- 不稳定，容易误判
- 无法稳定覆盖 `2714` / `02714.HK` / `AAPL` 等多种输入
- 后续维护成本高

## 决策

采用方案 B：

- 市场信息挂在每条股票持仓上
- 美股持仓走 Finnhub
- 港股持仓走 iTick

## 数据结构

### 股票持仓新结构

股票持仓新增 `market` 字段：

```json
{
  "code": "2714",
  "market": "HK",
  "shares": 100,
  "cashPosition": 0,
  "group": "其他"
}
```

### 价格缓存键

为避免不同市场同代码撞键，内部价格缓存统一改为“市场 + 代码”的组合键：

- 美股：`US:AAPL`
- 港股：`HK:2714`

UI 不直接展示这个缓存键，界面仍显示用户可读的代码与市场。

### 旧数据迁移

旧股票持仓如果没有 `market` 字段，按以下规则迁移：

1. 如果代码看起来像港股格式
   - 纯 4-5 位数字，例如 `2714`
   - 或形如 `2714.HK`
   - 或形如 `02714.HK`
   则迁移为 `market: "HK"`
2. 其他默认迁移为 `market: "US"`

对于迁移为港股的代码，再统一规范化为纯数字代码：

- `2714.HK` -> `2714`
- `02714.HK` -> `2714`
- `2714` -> `2714`

## 港股代码规范

### 用户可输入形式

港股录入时接受以下输入：

- `2714`
- `2714.HK`
- `02714.HK`

### 运行态规范化结果

统一存储为不带后缀、无前导零的纯数字字符串：

```text
2714
```

这样做的好处：

- 便于作为 iTick 的 `code` 参数直接使用
- 避免同一只股票因输入格式不同而被视为不同标的

## 数据源设计

### 美股

- 继续使用 Finnhub
- 请求参数沿用现有美股代码

### 港股

- 使用 iTick 的港股 REST API
- 请求形式为 `region=HK&code=<数字代码>`
- 当前优先使用单只报价或批量报价接口，取决于前端整合成本

### 路由规则

获取行情时，把股票持仓按市场拆成两组：

1. `US` 持仓 -> Finnhub
2. `HK` 持仓 -> iTick

两边请求完成后，把结果合并回统一的价格缓存对象。

## 界面设计

### 编辑页

股票持仓编辑与新增区域增加“市场”下拉：

- `US（美股）`
- `HK（港股）`

交互规则：

- 选择 `US` 时，代码输入提示为 `如 AAPL`
- 选择 `HK` 时，代码输入提示为 `如 2714 / 02714.HK`
- 保存时对港股代码做规范化处理

价格小标签不再固定为美元符号，而是按市场展示：

- `US` -> `$`
- `HK` -> `HK$`

### 账户详情页

每条股票持仓增加轻量市场标记，例如：

- `AAPL · US`
- `2714 · HK`

同时每条持仓名称旁边增加一个较小字号的占比文本，例如：

- `12.4%`

这个占比只需要是小数字提示，不需要单独的进度条或图表。

## 持仓占比计算

### 分子

单条持仓的当前市值。

- 股票账户：`shares * currentPrice`
- 基金账户：`marketValue`

### 分母

当前账户总体价值，包含：

1. 所有持仓当前市值之和
2. 当前账户全部流动现金之和

### 货币换算规则

持仓占比是无量纲的比例，只要分子和分母使用相同基准货币，结果就一致。

为了满足“港美股混合账户统一换算成美元后计算占比”的要求，规则定为：

1. 如果账户中同时出现 `US` 和 `HK` 股票持仓
   - 所有股票持仓与流动现金统一先换算成 `USD`
   - 再计算各持仓占比
2. 其他情况
   - 也统一使用一个单一基准货币计算
   - 为了实现简单和结果一致，股票账户可统一沿用 `USD` 作为占比计算基准
   - 基金账户沿用账户主币种即可

实现上，账户详情页只展示百分比，不展示该基准货币本身。

## 错误处理与兜底

### 单只股票报价失败

- 不影响其他股票报价更新
- 若本地已有旧价格，则继续使用旧价格
- 若从未拉到过价格，则显示为 `0`

### 港股数据源整体失败

- 页面仍可正常使用
- 美股刷新不受影响
- 港股本轮保持旧缓存价格

### API 限流

- iTick 免费层有调用限制
- 当前实现尽量优先采用批量接口，减少请求次数
- 若批量接口不可用，再退回单只请求

- 老持仓能自动迁移出 `market`

- 港股价格走 iTick，美股价格继续走 Finnhub
- 价格缓存键不会因不同市场撞码

- 账户详情页正确显示市场标记和小号占比文本
